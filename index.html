Description.

밥풀 프로젝트는 밥 약속 신청을 통해 관심사와 목표를 공유하는 사람들과 일대일로 대화할 수 있는 기회를 만들고자 했습니다. 긴밀한 대학생 커뮤니티, 서로 도움을 줄 수 있는 인맥 풀을 만들기 위한 플랫폼 입니다. SWYP 3기 프로젝트로 참여하여 1위를 수상 했습니다.

What did I do.
  • 실행계획 분석을 통한 페이징 응답 속도 98.77% 개선(73s → 900ms 이내)
    • 테스트 환경 조성을 위해 각 테이블에 100만 ~ 200만 건의 더미데이터를 삽입하는 과정에서 Batch Insert와 Bulk Insert 방식을 사용했습니다.
    • 페이지네이션 방식을 고수하되, 복합 인덱스 생성과 데이터 개수 기반의 ORDER BY & LIMIT 절을 사용해 최소한의 변경으로 최대 성능 개선을 이루어냈습니다. 이 과정에서 EXPLAIN 결과 분석과 복합 인덱스에 대한 이해를 높일 수 있었습니다.
    • 페이지 번호를 명시하기 위해 최적의 COUNT 집계함수 사용법을 찾아내고, 페이지에서 필요한 만큼의 최소 레코드 건수만 확인하도록 튜닝했습니다.
    • 블로그 링크 : 대용량 더미 데이터 넣고, 페이징 쿼리를 개선해보자
  • 락을 이용한 밥 약속 기능 동시 접근 제어
    • 네임드, 비관적, 낙관적락에 대해 학습 후 요구사항에 적합한 비관적 락을 MySQL FOR UPDATE 쿼리를 사용해 적용했습니다.
    • java.util.concurrent 패키지를 통해 멀티스레드를 이용한 테스트 코드를 작성하여 구현 코드의 신뢰성을 높입니다.
    • Jmeter 도구를 사용해 100명의 사용자가 동시에 같은 자원에 접근하였을 때 정합성이 지켜지는 것을 테스트했습니다.
    • 추후, 락에 의한 DB 부하를 방지하기 위해 Redis를 사용해 개선할 예정입니다.
    • 블로그 링크 : 밥 약속 기능으로 알아보는 동시성이슈 해결과정
  • 크롤링 봇 접근 제한 일 요청수 20,000 → 100건으로 축소
    • 크롤링 봇에 의해 AWS 과금이 발생하였고, 프로젝트 아키텍처와 비용을 고려해 문제를 해결했습니다.
    • Nginx 설정을 통해 특정 IP, 특정 키워드, 허용되지 않은 URL 차단 등을 적용했습니다.
    • 블로그 링크 : 일일 요청 수 2000건→110건, 봇 접근 제한하기
  • ID 채번 전략 비교 및 TSID 도입
    • Auto Increment 식별 값으로 자원에 대한 유추와 분산시스템 환경에서의 ID 생성 충돌을 고려해 ID 채번 전략을 개선했습니다.
    • Auto Increment, UUID v7, UUID v4, TSID 의 삽입 성능을 비교하여 TSID 생성 방식을 최종 도입했습니다.
    • TSID 식별 값이 Javascript 의 정수형 범위를 벗어나는 이슈를 JsonObjectMapper 를 커스텀하여 해결했습니다.
    • 블로그 링크 : ID 채번 전략 Trade-off 과정
  • JWT 로그인 방식 구현
    • 스프링 시큐리티를 사용하지 않고, 인증 과정에 필요한 필터/인터셉터 등을 직접 구현했습니다.
    • AccessToken의 만료기한을 짧게 생성하여 토큰 탈취로 인한 피해를 최소화하고, RefreshToken을 Redis에서 관리하여 보안과 성능을 타협했습니다.
    • API 마다 Request Header에 포함된 token을 추출해 사용자 식별 값을 가져오는 코드를 필터를 사용하여, 반복되는 코드를 제거하고 필요시 ~ 에서 간편하게 정보를 추출할 수 있도록 리팩터링 했습니다.
    • 사용자 권한에 따라 token 의 만료기한을 상이하게 생성하여, 개발 과정의 불편함을 최소화했습니다.
  • 클라우드 기반 애플리케이션 아키텍처 및 CI/CD 자동화 파이프라인 구축
    • 밥풀 프로젝트의 모든 아키텍처 적용과 CI/CD 스크립트 작성을 담당했습니다.
    • AWS 클라우드를 사용하여 프론트/백엔드가 분리된 아키텍처를 설계했습니다. VPC 내에 EC2 인스턴스는 Public subnet, RDS는 Private subnet에 추가하여 EC2로만 접근 가능하도록 설계했으며 이로인해 Public IP 과금 방지 효과도 가졌습니다.
    • Logback, Loki, Prometheus, Grafana 도구들을 활용해 모니터링 서버를 구축하여, 각종 지표 확인과 개발 직군의 원활한 커뮤니테이션에 이점을 가질 수 있었습니다.
